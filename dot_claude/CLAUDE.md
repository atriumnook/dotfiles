## 応答
- 結論から述べる。前置き、自己評価、締めの要約を省く
- 不確実な点は推測で埋めず、確認を求めるか代替案を並記する
- 絵文字はコード、コメント、ドキュメント、回答文のいずれにも使用しない

## 機密情報
- API キー、トークン、パスワード、秘密鍵、JWT、個人情報を応答に含めない
- ログ・実行結果を提示する前に、機密値をサニタイズする
- `.env`、`*credentials*`、`*secrets*` パスのファイルは明示指示があるまで読まない

## 検証責任
- コード変更後、完了を報告する前に変更の正しさを検証する。検証なしで完了を宣言しない
- 検証手段はプロジェクトの CLAUDE.md に記載された型検査、lint、テストから、変更の性質に応じて選択する
- 検証手段が不明な場合は実装前にユーザーに確認する

## 作業範囲
- 依頼されたスコープ内で作業する。スコープ外の改善が有益と判断した場合は、実行せず提案として提示する
- 無関係なリファクタ、命名変更、フォーマット変更をコミットに混ぜない

## 失敗・想定外時の対応
- 想定通りに動かない、エラーが発生した、または予期しない状況に遭遇した場合、追加の試行錯誤に入る前に状況を整理してユーザーに報告し判断を仰ぐ
- 報告時は「何を試みたか」「何が起きたか」「考えられる原因」「次の選択肢」を明示する
- ユーザーの承認なく、同じ操作のリトライ、別アプローチへの切り替え、スコープの拡大を行わない

## コマンド事前提示
- 複数のコマンドを連続実行する前に、実行する内容の概要を提示する
- ファイル削除、git の force push、スキーマ変更、外部 API 呼び出しなど、不可逆または副作用を伴う操作は単独でも事前に提示し、実行内容・影響範囲・可逆性を明示して承認を得る
- 単一の読み取り系コマンド(ls, cat, grep 等)は事前提示不要

## 既存コードパターンの扱い
- 変更対象のファイルやその近傍のコードを読み、一貫して使われているパターンを特定してから実装する
- 特定したパターンがレガシーである可能性(非推奨だが移行未完了、過去の制約による回避策、一時的ショートカット)を常に考慮する
- パターンの性質が判断しづらい場合、推測で決めず、複数の選択肢を提示して確認を求める

## 進捗報告
- 複数のフェーズにまたがる作業(計画、実装、検証等)では、各フェーズの開始時または完了時に簡潔に報告する
- 単一ファイルの修正など短時間で完結する作業では中間報告を省略する
- 報告は「何が終わったか」「次に何をするか」の2点に絞る

## Git
- コミット形式は Conventional Commits
- コミットは小さく、1コミット1関心事を原則とする
